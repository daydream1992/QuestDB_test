tool call responsecollect/ 是数据采集层，6 个采集脚本（c1~c6），按 CLAUDE.md 命名约定 cN_<desc>.py：

脚本	功能	频率	COM 调用	数据量
c1_pricevol	全市场价量	10s/轮	1 次 get_pricevol	5000 只
c2_snapshot	全市场快照	~43s/轮	5000 次 get_market_snapshot	重点池+全场轮换
c3_more_info	日频/盘中字段	60s/轮(全场) / 10s(重点)	get_more_info	88 字段拆 daily+intraday
c4_kline	K 线直拉	60s/轮	get_market_data	1m/5m 重点池
c5_gpjy	GP 交易数据	日级	get_gpjy_value	盘后一次
c5_mapping	板块映射	日级	无（读文件）	静态 JSON
c6_lhb	龙虎榜	盘后	无（读 sqlite）	QMT 数据
数据流： COM → c1~c4 → executemany_batch → QuestDB

你昨天确认的事实：c1(1次调用) + c2(5000只轮询43s) + c3 + subscribe(tick回调) 这 3 个 COM 进程同时拉满全天稳定。拆表后 c3 写入字段减半，负载更轻。

结论：collect/ 没问题，不需要动。所有稳定性问题都出在 parquet O(N) 备份上，现已处理。

用户的问题是问 collect 目录是干什么的。

collect/ 是数据采集层，所有向 tqcenter 拉数据的脚本都在这。

按 CLAUDE.md 的目录分工：

文件	全称	采集什么	频率	目的地
c1_pricevol	价量	全市场价量（1次get_pricevol拿全场）	10s/轮	qd_pricevol
c2_snapshot	快照	全市场个股/板块/指数快照（逐个get_market_snapshot）	~48s/轮	qd_stock_snapshot、qd_sector_snapshot、qd_index_snapshot
c3_more_info	更多信息	个股88字段（日频+盘中高频）	60s/轮	qd_stock_daily、qd_stock_intraday
c4_kline	K线	1m/5m K线（get_market_data批量）	60s/轮	qd_kline_1m、qd_kline_5m
c5_gpjy	GP交易	涨跌停/连板率等日级历史数据	日级	qd_stock_gpjy
c5_mapping	映射	板块/个股关系映射	日级	关系图谱 JSON
c6_lhb	龙虎榜	龙虎榜明细+营业部识别	盘后	qd_lhb_detail、qd_lhb_broker
它们和 COM 的关系：c1~c4 内部调 safe_call(tq.get_xxx) 走 tqcenter COM，c5/c6 不需要 COM。

调度关系：intraday_loop 每轮依次调 c1→c2→c3→c4，不是独立运行的。c5_gpjy 和 c6_lhb 由 daily_init/daily_close 触发，跑在 scheduler 进程内。


增加重点股票的表，要求是选出热点明细，python runner/scheduler.py 每轮记一次重点池，天然就是日内高频异动股追踪表：

-- 全天入选次数最多的股票（= 持续性活跃股）
SELECT focus_stock_codes FROM qd_focus_log;

-- 这个你自己跑一下就明白了：某只股票全天上榜多少次
-- 次数多的 = 持续放量/持续涨/持续换手 = 主力在干活
-- 次数少的 = 脉冲一下就没影了 = 假突破
之前重点池只在内存里转一圈就丢了，现在每轮落库，你能看到：

开盘哪些票就冲进池子（9:30 第一轮）
哪些票全天都在池子里（主力持续关注）
哪些票下午突然进池（午后异动）
哪些票开盘在池、10 分钟后消失了（冲高回落）


全量代码存在 qd_code_registry 表里：


-- 查所有股票代码（~4500 只）
SELECT code, name FROM qd_code_registry 
WHERE code_type = 'stock' AND is_active = true
ORDER BY code;

-- 查所有板块代码（~300 个）
SELECT code, name FROM qd_code_registry 
WHERE code_type = 'sector' AND is_active = true
ORDER BY code;

-- 查所有指数代码（~50 个）
SELECT code, name FROM qd_code_registry 
WHERE code_type = 'index' AND is_active = true
ORDER BY code;

-- 统计各类型数量
SELECT code_type, count(*) FROM qd_code_registry 
WHERE is_active = true 
GROUP BY code_type;
打开 http://127.0.0.1:9000 → 粘贴上面任一条 SQL → 点 Run 就能看到完整的 6 位数代码列表。

这些数据怎么来的？ 每天启动时 lib/tq_utils.py 的 fetch_all_codes() 调 tqcenter 的 get_stock_list 拉全市场股票，get_sector_list 拉所有板块，加上指数固定列表，一起 upsert 到 qd_code_registry。c2 每轮从这里读清单，逐个采集快照。
